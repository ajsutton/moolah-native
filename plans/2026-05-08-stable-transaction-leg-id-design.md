# Stable `TransactionLeg.id` — Design

**Date:** 2026-05-08
**Status:** Draft
**Author:** Adrian Sutton (with Claude)

## Problem

`TransactionLeg` (the domain model) has no `id` field. The leg's identity is generated by the repository on every write: each call to `create(_:)` and `update(_:)` allocates a fresh `UUID` per leg.

```swift
// Backends/GRDB/Repositories/GRDBTransactionRepository.swift
for (index, leg) in transaction.legs.enumerated() {
  …
  let legId = UUID()                              // ← fresh every time
  let legRow = TransactionLegRow(id: legId, domain: leg, …)
  try legRow.insert(database)
}
```

`update(_:)` deletes every existing leg row and re-inserts the new array, also with fresh ids. From CloudKit's point of view, every save of an existing transaction looks like *all-new leg records and a delete of every old leg record*. Even when the user only edits the payee, every leg's `record_name` flips.

This makes the system fragile in two ways:

1. **CK conflict resolution can't merge or dedupe legs.** A leg "moved" from one save to the next is a different record. The only thing tying two device-local writes together is the parent transaction id.
2. **Any path that drops a leg-delete uplink while landing leg-saves leaves orphan legs on the server**, which propagate back to the originating device on the next pull and surface as duplicates on later edits.

We have hit case 2 in production. Diagnosis (2026-05-08, against the Large Test profile imported from production data):

- 5 scheduled transactions had two identical legs (different leg UUIDs, same content). 0 of ~18,800 non-scheduled transactions did.
- Every paid copy of those scheduled transactions has 1 leg; only the *current* recurring instance has 2.
- The window of accumulation aligns with [`82d2f653`](https://github.com/ajsutton/moolah-native/commit/82d2f653) (explicit leg-cascade on delete; replaces the v3-era `ON DELETE CASCADE` that v5 dropped) and [`8f5279dc`](https://github.com/ajsutton/moolah-native/commit/8f5279dc) (leg-emit record-type regression — fixed for #681). Both fixes are now on main.

The two specific regressions are fixed, but the **class** of bug is still possible because legs have no stable identity. Any future regression that drops a leg-delete uplink will reproduce the same symptom.

## Decisions log

| Question | Decision |
|---|---|
| Should `TransactionLeg` gain a stable `id`? | Yes. |
| What is the source of the id? | The existing `transaction_leg.id` (`BLOB`/`UUID`) column. Domain just exposes it. |
| When is a fresh id allocated? | Only when a *new* leg enters the system: explicit "Add Sub-transaction", transfer counterpart insertion, trade fee append, `paidCopy(of:)` (paid copy is a new transaction with new legs), `applyAutofill` (replacing the draft from a match — new legs for a new write). |
| `update(_:)` semantics | Diff-by-id: legs in the new array that aren't in the old set get inserted; legs in the old set that aren't in the new array get deleted; legs present in both are upserted in place by id. No more "delete every old leg, re-insert every new leg with fresh UUIDs". |
| CKSync record mapping | No schema change. `record_name` already encodes the leg id (`TransactionLegRecord\|<UUID>`). The recordName for an existing leg is now stable across writes — which is the whole point. |
| Migration of existing data | None. Existing GRDB rows already have stable ids in `transaction_leg.id`; the row→domain mapper currently throws them away. After the change, the mapper preserves them. |
| Existing in-the-wild duplicates | Out of scope; user will clean up manually. (Not adding a one-shot dedup utility.) |
| Backwards compat for older record names? | Not needed — `record_name` shape is unchanged. The only difference is that `update` no longer churns the recordName. |

## Section 1 — Domain model change

`TransactionLeg` becomes:

```swift
struct TransactionLeg: Codable, Sendable, Hashable, Identifiable {
  let id: UUID                          // NEW — stable across writes
  let accountId: UUID?
  let instrument: Instrument
  let quantity: Decimal
  let externalId: String?
  let counterpartyAddress: String?
  var type: TransactionType
  var categoryId: UUID?
  var earmarkId: UUID?
}
```

`Identifiable` conformance is incidental but useful for `ForEach(transaction.legs)`.

The `id` is `let` because a leg's identity must not change after creation. All existing mutating fields (`type`, `categoryId`, `earmarkId`) keep their `var` declarations. `accountId`, `instrument`, `quantity`, `externalId`, `counterpartyAddress` stay `let` — those changes are conceptually a "different leg" today and may want explicit normalisation later, but that's not in scope here.

### Where new ids are allocated

A fresh `UUID()` is allocated at exactly the points where a leg first enters the system. Existing call sites that construct `TransactionLeg` get an explicit `id:` argument; the default value defers to `UUID()`:

```swift
init(
  id: UUID = UUID(),
  accountId: UUID?,
  instrument: Instrument,
  quantity: Decimal,
  externalId: String? = nil,
  counterpartyAddress: String? = nil,
  type: TransactionType,
  categoryId: UUID? = nil,
  earmarkId: UUID? = nil
)
```

Most callers (test fixtures, CSV import, crypto importer, `Transaction.default*`) take the default and never name the id — semantics are unchanged for them.

A few specific call sites must consciously *not* preserve an upstream id:

- `Transaction.paidCopy(of:)` — the paid copy is a new transaction. Its legs must get **new** ids. Implementation:
  ```swift
  static func paidCopy(of scheduled: Transaction) -> Transaction {
    Transaction(
      id: UUID(),
      date: scheduled.date,
      payee: scheduled.payee,
      notes: scheduled.notes,
      legs: scheduled.legs.map { TransactionLeg(id: UUID(), …) }   // copy, new ids
    )
  }
  ```
- `TransactionDraft.toTransaction(id:)` — currently constructs `TransactionLeg`s from `LegDraft`s. After the change, `LegDraft` carries a `legId: UUID?` (nil for "this leg is new in the draft"). `toTransaction` uses the existing id when present, fresh `UUID()` otherwise.

### `LegDraft` carries the leg id

```swift
extension TransactionDraft {
  struct LegDraft: Sendable, Equatable {
    let legId: UUID?                      // NEW — nil for legs added in-draft
    var type: TransactionType
    var accountId: UUID?
    var amountText: String
    var categoryId: UUID?
    var categoryText: String
    var earmarkId: UUID?
    var instrument: Instrument?
  }
}
```

`legId` is `nil` for legs created within the draft (`addLeg`, the transfer counterpart `insert(at:)` in `setType`, the trade `appendFee`, the trade `switchToTrade` `[blank, carried]` pair, etc.) and is populated when the draft is initialised from an existing `Transaction`. `toTransaction(id:)` resolves `legId ?? UUID()` per leg.

`legId` is intentionally `let`. A leg's identity persists through every edit the user makes in the inspector. The mode-switch helpers (`setType`, `switchFromTrade`) preserve `legId` on legs they carry forward and leave it nil on legs they construct fresh.

**Exception — `switchToTrade`.** Both legs produced by `switchToTrade` (`carried` and `blank`) get `legId: nil`, including the one carrying the pre-trade leg's amount text. The reasoning: trade mode replaces a single income/expense/transfer leg with a *pair* (paid + received) on the same account, with negated-sign accounting and structural constraints `setType` cannot model. The "carried" leg is no longer the same row in any GRDB sense — it has different accounting semantics, a different position in the leg array, and is paired with a sibling that didn't previously exist. Treating both pair members as new legs (and accepting one extra leg-delete + leg-insert on the next save) is the simpler, less surprising contract than half-preserving identity through a structural transformation. `switchFromTrade` is symmetric in the other direction; it likewise constructs new simple-mode legs rather than trying to thread identity back from the trade pair.

## Section 2 — Repository update semantics

The current `update(_:)` deletes every old leg and re-inserts every new one with fresh ids. The new `update(_:)`:

```
old_ids = SELECT id FROM transaction_leg WHERE transaction_id = T.id
new_ids = T.legs.map(\.id)

deleted = old_ids \ new_ids                       # legs the user removed
upserted = T.legs                                 # everyone else (insert-or-update by id)

DELETE … WHERE id IN deleted
for leg in upserted: legRow.upsert(database)
```

`upsert` is idempotent for legs whose content hasn't changed (the row equals what's already there), so calling it for every leg in the array is safe and removes the need to detect "did this leg actually change". The cost is one row-write per leg per save, same as today.

`UpdateOutcome` continues to expose `deletedLegIds` and `insertedLegIds` for the post-commit hook fan-out. After the change `insertedLegIds` becomes "upserted leg ids" (renamed for clarity); the sync hook upload is identical (CK upsert is also idempotent). `onRecordChanged(TransactionLegRow.recordType, legId)` fires for every leg in the new array on every update — no behaviour change for the sync layer beyond the absence of the spurious "leg X was deleted, leg X' was inserted" pair when nothing actually moved.

## Section 3 — Wire / sync layer

**No schema change.** `transaction_leg.id` already exists as the primary key; the domain change just stops discarding it on read.

**`record_name` becomes stable across writes.** Today, every `update(_:)` invents a fresh `record_name` (`TransactionLegRecord|<NEW-UUID>`) for every leg. After the change, the leg's `record_name` is set once at creation and reused on every subsequent save. This is the property that closes the orphan-on-server-after-uplink-failure failure mode: a re-pulled leg from the server upserts onto the existing local row instead of creating a phantom duplicate.

**`encoded_system_fields`** stays per-row. After the first successful save the leg's CKRecord system fields are cached locally; subsequent saves attach the same blob, so CK's modification version vector keeps working unchanged. **Implementation note (load-bearing):** GRDB's `upsert(_:)` resolves to `INSERT … ON CONFLICT DO UPDATE SET …`, which writes every column from the new struct — including `encodedSystemFields`, which `TransactionLegRow.init(domain:transactionId:sortOrder:)` hardcodes to `nil`. The diff-by-id update path therefore must fetch the existing rows and re-attach their `encodedSystemFields` blob before each upsert. Without that, every header-only edit (e.g. renaming a payee) would zero the change-tag on every leg in the transaction, causing `unsyncedRowIdsSync()` to flag them as unsynced and triggering a spurious re-upload of every leg on every save. Reference R1 in the implementation plan covers this; the per-leg fetch is one extra round-trip on the writer queue, amortised across the user's typing cadence.

**`applyRemoteChangesSync`** is already correct (`row.upsert(database)` keyed on `id`). The fact that two devices independently writing legs with two different ids could still produce duplicates would *technically* still be possible, but the only paths that allocate new ids are the four enumerated above (genuine new-leg creation), and a "different new leg with the same content" is exactly the human writing two separate legs that happen to look the same. That's not a bug to fix at this layer.

**Note on GRDB zombie legs.** If CloudKit delivers a `TransactionLeg` save in the same fetch batch as a parent `TransactionRecord` deletion (or in a prior batch), the leg upsert may land first via `GRDBTransactionLegRepository.applyRemoteChangesSync`, and the transaction-deletion cascade in `GRDBTransactionRepository.applyRemoteChangesSync` cleans the orphan up in the same or a later batch (it deletes legs `WHERE transaction_id = ?` regardless of whether the transaction-level id was ever resident on this device). This matches the zombie-row contract in `guides/SYNC_GUIDE.md §3`; such rows do not surface in views because `GRDBTransactionRepository+Fetch` joins legs to their parent. This is a different scenario from the CloudKit-server orphan-leg case that this change closes.

## Section 4 — Test plan

**Domain layer**

- `TransactionLegEqualityTests`: legs with same content but different `id` are not `==`. (Reason: `Hashable`/`Equatable` derives field-by-field, and id is now part of the struct.)
- `TransactionLegPaidCopyTests`: `paidCopy(of:)` returns a transaction whose legs have ids *different* from the source's. (Otherwise the copy would collide with the original on save.)

**Repository layer**

- `GRDBTransactionRepositoryUpdatePreservesLegIdsTests`: create transaction with two legs, mutate one leg's `categoryId`, save. Re-fetch. Both legs' ids are unchanged. (Plan: Stage 4 `headerOnlyEditPreservesLegIds`.)
- `GRDBTransactionRepositoryUpdateAddRemoveLegsTests`: start with two legs, add a third, save. Re-fetch. The two original ids are unchanged; the new leg has a new id. (Plan: Stage 4 `removeOneLegPreservesSurvivor` covers the remove case; the add-a-leg case is structurally identical and can be folded into the same suite.)
- `GRDBTransactionRepositoryUpdateRemoveLegEmitsDeleteHookTests`: removing a leg from the array triggers `onRecordDeleted(TransactionLegRow.recordType, removedId)` and *no* delete hook for the legs that stayed. (Plan: Stage 8 `transactionUpdateEmitsLegRecordType` asserts `legDeletes.isEmpty` for the preserve-ids path; the remove-leg-emits-one-delete case follows from `transactionUpdateWithReplacedLegsEmitsDeletes` adapted for a remove-only scenario.)
- `GRDBTransactionRepositoryUpdateNoSpuriousLegChurnTests`: edit only the parent transaction's `payee`, no legs touched. Assert that the `onRecordChanged` calls for legs are still emitted (legs are upserted) but `onRecordDeleted` is empty. Document the reason for the upsert: it's cheap and lets the sync layer rebuild the `transaction_leg` `record_name` set if it's been corrupted. (Plan: covered by Stage 4 `headerOnlyEditPreservesLegIds` for ids and Stage 8 `transactionUpdateEmitsLegRecordType` for hook fan-out shape — no separate test file is needed.)
- `GRDBTransactionRepositoryUpdateRoundTripStabilityTests`: load txn → modify a leg's category through `Transaction(legs: …)` → save → reload → leg ids identical, content updated. (Plan: Stage 4 `headerOnlyEditPreservesLegIds`.)
- `GRDBTransactionRepositoryUpdateEncodedSystemFieldsPreservedTests`: header-only edit must not zero the cached `encodedSystemFields` blob on legs that survived the upsert. (Plan: Stage 4 `headerOnlyEditPreservesEncodedSystemFields`.)
- `GRDBTransactionRepositoryUpdateRollbackTests`: a thrown error inside `performUpdate` (e.g. `ensureInstrumentReadable` rejecting an unknown instrument id) leaves both header and legs in their pre-update state. (Plan: Stage 4 `performUpdateRollsBackOnFailure`.) Required by `guides/DATABASE_CODE_GUIDE.md §5` for any multi-statement write.

**Sync layer**

- `TransactionLegSyncPushIdempotenceTests`: simulate `applyRemoteChangesSync(saved: [legRow])` for a leg already in GRDB with the same id. No duplicates created. (Plan: Stage 8a `sameLegIdUpsertIsIdempotent`.)
- `TransactionLegSyncOrphanReintroductionTests` (regression test for the production symptom): load a txn with one leg, simulate the sync engine delivering a phantom leg row with a UUID the local row does NOT share (the server-side orphan case). The phantom lands as a second row — documenting the residual race that the design knowingly leaves for manual cleanup. (Plan: Stage 8a `differentIdOrphanLandsAsSecondRow`.)

  *Note*: the orphan-reintroduction test pins **behaviour**, not a guarantee. It documents that the repository upserts by id, so a server-side orphan with a different id can still be applied — that's the user's manual cleanup case described in Section 7.

**Driver coverage** (UI, optional but recommended)

- A `MoolahUITests_macOS` regression test extending the existing transaction-detail driver: open a multi-leg transaction in the inspector, edit a single leg's amount, save, reopen — assert the leg-row order and content match. (Today's behaviour is the same; the driver test guards against future changes that might re-randomise leg order on save.)

## Section 5 — Compatibility and rollout

**Reading existing data.** The first time a user runs the new code, every leg in `transaction_leg` already has a stable id in the `id` column. The row→domain mapper picks it up. No migration step. No data conversion.

**Writing.** The first update after upgrade differs from the old behaviour: instead of "delete all, re-insert with fresh ids" the new code sees that every leg in the new array has the same `id` as the existing row in `transaction_leg`, so it upserts in place. The recordName-stabilisation kicks in on this very first save — no transitional delete-then-recreate phase.

**Rollback.** Reverting the change is straightforward — the domain `id` field is additive (default-initialised on construction), the repository diff logic is reversible to "delete all + re-insert", and `transaction_leg.id` is unchanged in the schema.

## Section 6 — Open questions

1. **Should `LegDraft.legId` be observable for inspector tests?** Probably not — tests should assert on content, not id. Listed here in case the implementation surfaces a need.
2. **Do we expose `Transaction.legs` mutation as add/remove APIs that allocate ids automatically?** I lean yes, as a small follow-up: `var legs: [TransactionLeg]` works fine, but a `mutating func appendNewLeg(…)` would prevent callers from forgetting to set a fresh id. Out of scope for this design; revisit when implementing.
3. **Crypto importer and CSV importer**: both currently construct `TransactionLeg`s with no id concern. They keep working because the default `UUID()` argument means new legs always get new ids. Existing `externalId`-keyed dedup paths (the crypto deduper) are unaffected.

## Section 7 — Out of scope

- **Cleaning up the existing 5 duplicate legs in production data.** User will fix manually.
- **Adding a generic dedup pass** for non-`externalId` legs. The class of bug that produced them is closed by the recordName-stability change; a generic dedup pass introduces new "what counts as a duplicate?" decisions and has not been requested.
- **Changing `TransactionLeg`'s mutability model** (e.g. making `accountId` `var`). Possibly worth doing later but unrelated to identity.
- **Restructuring `applyRemoteChangesSync` to compare with local state before upserting.** The current "upsert by id" is correct under the new identity rules.

## Section 8 — Acceptance criteria

For the implementation plan to be considered complete:

- `TransactionLeg.id: UUID` exists, is `let`, and is preserved end-to-end across `update(_:)`.
- `record_name` is stable for the lifetime of a leg (verified by a sync-layer test).
- All `TransactionRepository` contract tests pass.
- All inspector / store tests pass with no behavioural change to user-visible flows.
- A new regression test pins the "edit-only-the-payee, leg ids don't change" property.
- No new SwiftLint baseline entries introduced.
- `code-review`, `concurrency-review`, `database-code-review`, `sync-review` agents pass on the diff.
