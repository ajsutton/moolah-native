# Reviewing UI With Previews

Tight visual iteration on a single SwiftUI view, driven from the conversation. The user watches the Xcode canvas; you render the same preview to PNG via the Xcode MCP and reason about it; both of you converge on a design without leaving Xcode and without rebuilding the whole app.

## When this skill applies

- User asks to iterate on or polish a specific view ("let's clean up the Create Account dialog", "the picker popover looks weird").
- User wants to give visual feedback on a layout change you propose.
- User mentions Xcode previews, the canvas, or `#Preview`.
- User mentions a UI bug in a single view (spacing, focus order, missing label, popover behaviour) where the fix is local to the view file.

## When this skill does NOT apply

- The work spans multiple windows or deep navigation that previews can't render (e.g. push from sidebar → list → detail → sheet). Use `automate-app` or `run-mac-app-with-logs` instead.
- The work needs real CloudKit sync, network requests, or other live backends. Previews use `PreviewBackend` only.
- The user wants to verify behaviour under real user input across many screens. Use a UI test (`writing-ui-tests`) or the running app.

## Setup

### 1. Worktree

UI iteration is a code change — `main` is protected. Use the `superpowers:using-git-worktrees` skill to create a feature branch:

```bash
git -C /Users/aj/Documents/code/moolah-project/moolah-native worktree add \
  .worktrees/ui-<view-name> -b ui/<view-name>
```

### 2. Generate the Xcode project in the worktree

`Moolah.xcodeproj` is gitignored and regenerated by `xcodegen`. After creating the worktree:

```bash
just --justfile <worktree-path>/Justfile --working-directory <worktree-path> generate
```

Note the `--justfile` + `--working-directory` form — `just -d <path>` alone fails with "the following required arguments were not provided: --justfile <JUSTFILE>".

### 3. Open Xcode at the worktree

```bash
open -a Xcode <worktree-path>/Moolah.xcodeproj
```

### 4. Confirm the right window is active

```
mcp__xcode__XcodeListWindows
```

You'll get back a list with `tabIdentifier` per open project. Identify the worktree's tab (its workspacePath ends in `<worktree-path>/Moolah.xcodeproj`). All later MCP calls (`XcodeRead`, `RenderPreview`, etc.) need that `tabIdentifier`. If the user already had a `main` checkout open, both will be listed.

### 5. Open the target view file

```bash
open -a Xcode <worktree-path>/<path-to-view>.swift
```

### 6. Verify the preview renders

```
mcp__xcode__RenderPreview \
  tabIdentifier=<worktree-tab> \
  sourceFilePath=<relative-path-to-view> \
  timeout=300
```

Returns a PNG path. `Read` it to see the snapshot. If the first call returns `ResourceInvalidationError` ("cancel update group" / "RichCancelationError"), retry once — the preview infrastructure sometimes cancels its first build mid-flight when SourceKit indexing is still settling.

### 7. Tell the user to open the canvas

Ask them to enable the canvas in their Xcode window (⌥⌘↩) and click Resume (⌥⌘P) if the preview isn't already live. Once their canvas shows the same image you just rendered, you're synced.

## The iteration loop

Each round:

1. Make a focused edit to the view file (or a closely-related shared view) using `Edit`.
2. Call `mcp__xcode__RenderPreview` on the same `tabIdentifier` and `sourceFilePath`.
3. `Read` the returned PNG.
4. Describe what changed and what new issues are visible. Be specific about row positions, label text, missing elements, etc. — the user is looking at the same image.
5. Ask the user to confirm or redirect.

If the render fails with `ResourceInvalidationError` after a code edit, retry the render once before reporting a failure — Xcode often cancels the in-flight render when the file changes mid-build.

If the render shows compile errors (visible as a red X in the snapshot or as `new-diagnostics` system reminders pointing at lines you just touched), fix the syntax error before re-rendering. Stale `Cannot find type 'XYZ'` diagnostics from SourceKit indexing are noise — treat as ignorable unless they reference a symbol you just renamed.

## Interaction the preview supports

The user can interact directly with the running preview in their canvas:

- **Click** buttons → triggers actions (sheets/popovers will present, sometimes with caveats — see below).
- **Type** into focused text fields.
- **Open popovers** — popovers presented from a button inside the previewed view DO render and respond to clicks. They render in the canvas, not in a separate window. You can iterate on popover content this way.
- **Selectable view mode** — Xcode's "Selectable" preview mode (toolbar at the bottom of the canvas) draws a selection outline around any view, exposing the actual rendered bounds. Use this when the user reports "lots of white space" or "the dialog is too tall" to disambiguate canvas size from view size.

What the static preview snapshot **cannot** show:

- `.sheet(isPresented:)` content — sheets need a real window context. The preview will render only the host. To iterate on sheet content, render the sheet view's `#Preview` directly (don't try to wrap in `.sheet`).
- `navigationTitle` and toolbar chrome — these only render when there's a real window/sheet host. The title bar in a static preview shows "Xcode Previews" instead. Don't chase this — it works in the running app.
- Real sheet sizing on macOS — `.frame(minWidth:minHeight:)` on a sheet's root view is treated as the floor when presented as a real sheet, but the preview canvas can be arbitrarily large and the view fills it. Don't try to "fix" white space below a form in the preview if it's actually a canvas-vs-sheet artefact. Ask the user what they see in Selectable mode before chasing it.

## Do NOT launch the running app

This is the user's stated preference for UI iteration: **do not run `just run-mac` between edits.** Build cycles burn time and context, and the preview already shows real interaction behaviour for nearly all view-local work. The user can interact with popovers, type into fields, and exercise focus directly in the canvas.

Only fall back to the running app when:

- The user explicitly asks ("launch the app so I can try X").
- The change is below the view layer (store/repo/sync) and won't reflect in a preview.
- The interaction needs a real sheet host that the preview can't render.

If you think you need the running app, **ask first.** Don't speculatively rebuild.

## Common stumbling points

### `Color(nsColor:)` and other AppKit symbols in `#Preview`

The view file is shared between iOS and macOS targets, so any preview content you add must compile for both. `Color(nsColor:)`, `NSColor.*`, etc. break iOS builds. Use cross-platform colours (`Color.gray`, `Color.accentColor`, etc.) or wrap in `#if os(macOS)`.

### `.datePickerStyle(.compact)` does not visibly change the date picker on macOS in a Form

In `.formStyle(.grouped)` on macOS the default DatePicker is the stepper-field style ("26/ 4/2026 ⌃⌄"). `.compact` is silently overridden in this context. The stepper style is intentional in this project — the up/down arrow keys are a deliberate keyboard affordance. See `feedback_keep_stepper_date_picker.md` in user memory.

### Popover focus is its own problem

`@FocusState` and `.defaultFocus(_:_:)` inside a `.popover(...)` content are full of platform quirks. Read `guides/FOCUS_GUIDE.md` before changing anything related to focus. Do not invent timing-based workarounds (`Task.sleep`, `DispatchQueue.main.asyncAfter`) — the user has called these out as unacceptable.

### Stale SourceKit "Cannot find type" diagnostics

Right after `just generate` and Xcode opens the regenerated project, SourceKit's index has not finished. You'll see `Cannot find type 'AccountStore' in scope`-type diagnostics from `XcodeRefreshCodeIssuesInFile` for files that compile fine. These resolve on their own; ignore them unless they reference something you just renamed.

### Codesign `.cstemp` errors

`Sign Moolah.app: ... invalid or unsupported format for signature ... .cstemp` blocks `RenderPreview` from rebuilding. A `codesign` run was interrupted (often by killing the running app while Xcode was building it). Fix:

```bash
find /Users/aj/Library/Developer/Xcode/DerivedData/Moolah-*/Build/Products \
  -name "*.cstemp" -delete
```

Then re-render. If it persists, ask the user to do **Product → Clean Build Folder** (⇧⌘K) on the worktree's Xcode window.

## What to commit when done

The skill produces real production changes — the only artefacts that should land are:

- The view file edits you iterated on.
- Any new shared helpers/properties you added (e.g. picker display formatters).
- Tests for any new shared logic — if you added a property like `Instrument.pickerLabel`, add a unit test for it.

Do **not** commit:

- Preview-only `.frame()`, `.background()`, `.sheet(host)` wrappers added during iteration. Strip them before committing.
- Debug print statements.
- Imports added solely for preview experiments.

## Reference: tools used in this loop

| Tool | When to use |
|------|-------------|
| `Edit` (on the view file) | Apply a focused change. |
| `mcp__xcode__RenderPreview` | Snapshot the current state. |
| `Read` (on the returned PNG) | See what the user sees. |
| `mcp__xcode__XcodeListWindows` | Confirm which Xcode window is the worktree. |
| `mcp__xcode__XcodeRead` | Read a file as Xcode sees it (preserves indentation). |
| `Bash` (`open -a Xcode <path>`) | Open a file or project in Xcode. |
